# G0 봉인 런북

G0 = "hermes 실물 `pre_tool_call` 페이로드가 플러그인 hook 계약과 일치한다"를 증명하는 최소 봉인 단계.

---

## ✅ 봉인 결과 (SEALED — hermes v0.16.0)

설치된 hermes(`~/.local/bin/hermes`, **v0.16.0**, 2026.6.5)의 소스
(`/tmp/hermes-src`)에서 **실제 호출부**를 확인하고, 실제 플러그인 코드가 그 kwargs를
정확히 처리함을 검증해 봉인 완료.

**실제 plugin pre_tool_call 계약** — 출처: `hermes_cli/plugins.py`
`get_pre_tool_call_block_message()` → `invoke_hook("pre_tool_call", ...)`:

```python
invoke_hook("pre_tool_call",
    tool_name=...,            # str
    args=args if isinstance(args, dict) else {},   # 항상 dict
    task_id=..., session_id=..., tool_call_id=..., turn_id=...,  # 기본값 ""
    api_request_id=..., middleware_trace=[...])    # 플러그인 미사용 → **kwargs 흡수
```

**핵심 발견**:
- hermes는 pre_tool_call 플러그인 hook에 **`cwd`/`git_branch`를 전달하지 않는다.**
  플러그인은 이를 optional로 처리(부재 시 이벤트에서 `null`) → 계약 일치. ✅
- id 기본값은 빈 문자열 `""` → 플러그인이 `null`로 정규화. ✅
- `api_request_id`/`middleware_trace`(+미래 키)는 `**kwargs`로 흡수, hook은 throw 없음. ✅
- 경로는 realpath 정규화(심볼릭링크 해소)되어 이벤트에 기록. ✅

**shell hook ≠ plugin hook**: `hermes hooks test`는 cli-config.yaml의 **shell hook**만
발화시킨다(아래 LIVE 절차). SyncSpace는 **Python 플러그인(register_hook)**이라 그 경로로는
캡처되지 않으며, 위 invoke_hook 호출부가 권위 있는 계약이다.

**회귀 고정**: `tests/test_g0_seal.py` — 실제 kwargs를 `on_pre_tool_call`에 흘려보내
이벤트 형태(action/tool/paths/세션·턴·correlation 매핑, cwd·git_branch=null, 추가 kwargs 흡수)를 검증.

## G0 봉인이란

플러그인은 hermes 가 `invoke_hook("pre_tool_call", ...)` 로 전달하는 kwargs 중 아래 필드에 의존한다.

| 필드 | 필수 여부 | 플러그인 사용처 |
|---|---|---|
| `tool_name` | **필수** | `on_pre_tool_call` 첫 인자, `extract_raw_paths`, `_map_action` |
| `args` | **필수** | `extract_raw_paths`, `_build_detail` — dict 여야 경로 추출 가능 |
| `session_id` | 선택 | `derive_agent_id`, 규칙 scope 매칭 |
| `task_id` | 선택 | 이벤트 `taskId` |
| `turn_id` | 선택 | 이벤트 `turnId` |
| `tool_call_id` | 선택 | 이벤트 `correlationId` |
| `cwd` | 선택 | `normalize_path` base, 이벤트 `cwd` |
| `git_branch` | 선택 | 이벤트 `gitBranch` |

"봉인(sealed)" = 실물 hermes 가 위 필수 필드를 포함한 페이로드를 실제로 전달함을 `g0_captured_payload.json` 아티팩트로 증명한 상태.

## LIVE 캡처 절차 (hermes 설치 후)

### 사전 조건

- `hermes` CLI 가 PATH 에 있을 것 (`which hermes` 로 확인)
- hermes 설정 파일(`~/.hermes/cli-config.yaml`)이 존재할 것

### 단계

**1. 캡처 셸 훅 작성**

```bash
mkdir -p ~/.hermes/agent-hooks

cat > ~/.hermes/agent-hooks/g0_capture.sh << 'EOF'
#!/bin/bash
# G0 캡처 훅 — stdin JSON 을 파일로 저장
OUTDIR="$(cd "$(dirname "$0")" && pwd)"
OUTFILE="$OUTDIR/g0_live_payload.json"
cat > "$OUTFILE"
echo "[G0] 캡처 완료: $OUTFILE" >&2
EOF

chmod +x ~/.hermes/agent-hooks/g0_capture.sh
```

**2. hermes 설정에 훅 등록**

`~/.hermes/cli-config.yaml` 에 추가:

```yaml
hooks:
  pre_tool_call:
    - matcher: "write_file|patch|read_file|search_files"
      command: ~/.hermes/agent-hooks/g0_capture.sh
      timeout: 10
```

비대화형 동의: `export HERMES_ACCEPT_HOOKS=1`

**3. LIVE 캡처 실행**

```bash
HERMES_ACCEPT_HOOKS=1 python3 scripts/g0_capture.py --live
```

또는 직접:

```bash
hermes hooks test pre_tool_call --for-tool write_file
```

**4. 검증 실행**

```bash
python3 scripts/g0_validate.py scripts/g0_captured_payload.json
```

**5. read/write/patch/search_files 각각 샘플 수집** (권장)

```bash
hermes hooks test pre_tool_call --for-tool read_file
hermes hooks test pre_tool_call --for-tool write_file
hermes hooks test pre_tool_call --for-tool patch
hermes hooks test pre_tool_call --for-tool search_files
```

각 출력을 `scripts/g0_live_<tool>.json` 으로 보관하면 G0 아티팩트 완성.

## 검증기 출력 해석

```
PASS  [tool_name] = 'write_file'
PASS  [args] dict(['path', 'content'])
PASS  [args.path] = '/tmp/test.txt'
PASS  [session_id] = 'sess-abc123'
...
------------------------------------------------------------
[G0 VALIDATE] PASS — 모든 필수 + 선택 필드 존재. G0 봉인 준비 완료.
```

| 출력 패턴 | 의미 |
|---|---|
| `PASS [field]` | 필드 존재하고 타입 유효 |
| `WARN [field] 없음` | 선택 컨텍스트 키 부재 — 이벤트 품질 저하 가능, 차단 동작은 정상 |
| `FAIL [field] 필드 없음` | 필수 필드 누락 — 플러그인이 경로 추출/규칙 평가를 할 수 없음 |
| `PASS (경고 N개)` | 봉인 가능 — 선택 필드 채우면 이벤트 품질 향상 |
| `FAIL — 필수 필드 N개 누락` | 봉인 불가 — hermes 페이로드 형식 변경 가능성, 재확인 필요 |

종료 코드:
- `0` — 필수 필드 모두 통과 (경고 있어도 0)
- `1` — 필수 필드 하나 이상 누락
- `2` — 파일 읽기/파싱 실패
